= How Puppet integrates with {Project}

[role="_abstract"]
Puppet uses a server-agent architecture.
An OpenVox server communicates with an OpenVox agent to manage the configuration of hosts.
{Project} components integrate with that architecture to provide configuration management capabilities in {Project}.

Puppet architecture in {Project} comprises the following components:

OpenVox server::
OpenVox server is the central component that stores Puppet configuration definitions in {Project}
It is deployed on {ProjectServer} or {SmartProxyServers}.
+
[NOTE]
====
Puppet server is no longer compatible with {Project} for Puppet integration.
To integrate your Puppet infrastructure with {Project}, you need to use OpenVox server instead.
OpenVox server is functionally equivalent to Puppet server.
====

OpenVox agent::
An OpenVox agent is a service that communicates with OpenVox server to manage the configuration of hosts.
It is installed on {Project} hosts.
+
[NOTE]
====
As a legacy alternative, Puppet agent remains compatible with {Project} for Puppet integration.
OpenVox agent is functionally equivalent to Puppet agent.
====

Puppet modules::
A Puppet module is a collection of classes, manifests, resources, files, and templates.
The Puppet modules work as components of host configuration definitions.

Puppet environments::
You can create multiple Puppet environments to control versions of configuration definitions or to manage variants of the definitions, and to test the definitions before you deploy them on production.

On each run, the OpenVox agent collects Puppet facts about the host and sends them to OpenVox server.
OpenVox server forwards those facts to {Project}.

[NOTE]
====
To list Puppet facts on a host, run the `puppet facts` command.
====

{Project} acts as an external node classifier (ENC) for OpenVox server.
Based on the facts and other definitions {Project} receives from OpenVox server, {Project} constructs the ENC answer to OpenVox server.

Based on the ENC answer, OpenVox server compiles a catalog for the host and sends it to the agent.

The agent evaluates the system state on the host.
If the agent finds differences between the desired state defined in the catalog and the actual state, also known as drifts, it enforces correction of the state of the host.

The agent reports correction results back to OpenVox server.
OpenVox server receives correction results from the agent and reports them to {Project}.

If the Puppet modules you are using support the use of parameters, you can override parameters of the module by using Smart Class parameters in {Project}.
In {Project}, you define the parameters as key-value pairs, which behave similar to host parameters or Ansible variables.
