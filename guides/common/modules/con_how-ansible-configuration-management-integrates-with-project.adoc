= How Ansible configuration management integrates with {Project}

[role="_abstract"]
Ansible roles describe how a host should look over time, unlike one-off remote execution jobs that only perform immediate tasks.

Configuration management by using Ansible and Ansible roles is for long-term state management (configuration management).
As opposed to remote execution, which is for ad hoc tasks and orchestration.
When you use Ansible roles for configuration management, you use roles as a way to define what a host should look like.

Persistence::
You assign roles to a host or host group in {Project}.
They become part of the host's definition.

Additionally you can override Ansible variables directly in the {ProjectWebUI}, allowing you to customize roles per host or host group.

The goal is to ensure the host maintains a specific configuration over time.

[NOTE]
====
While running an Ansible role on a host triggers a remote execution job in the background, there is a difference in the use case between configuration management and remote execution.
When you run remote jobs with remote execution, you trigger those jobs as needed.
They do not define the host's required permanent configuration.
An Ansible role defines what the host is or should be.
Running a remote execution job ad hoc defines what the host should do now.
====
